Introduction

Wodel-EDU extends Wodel with the automated generation of educational exercises. A complete Wodel-EDU specification is written with a family of five DSLs, each in its own file within the project:

  • Wodel (.mutator): defines the mutation operators, organised in blocks, that produce the incorrect variants of the seed models.
  • eduTest: configures the questionnaire and defines the exercises, selecting the exercise type of each assignment and features such as navigation, reattempts, penalties or option ordering.
  • modelDraw (optional): configures the graphical visualisation of the models, rendered with Graphviz, Circuit Macros or PlantUML.
  • modelText (optional): configures the textual representation of the model elements.
  • mutaText (optional): configures the text that describes the applied mutations, used to build the options of the text-based exercises.

This page is a hands-on tutorial covering the four Wodel-EDU DSLs construct by construct; the Wodel mutation language itself is covered in the Wodel tutorial. As in the rest of the site, the examples use the finite automata metamodel, whose main classes are Automaton, State (with attributes name, isInitial and isFinal), Transition (with references src, tar and symbol) and Symbol.

1. The big picture: from mutations to exercises

Wodel-EDU derives families of exercises automatically by mutating seed models. The correct and mutated models become the alternatives of multiple-choice questions, the applied mutations become the emendation options of correction questions, and their textual renderings feed matching-pairs, missing-words and drag-and-drop exercises. The connection between the DSLs is the Wodel block: the Wodel program groups its mutations into named blocks, and each eduTest exercise references the block(s) whose mutants it should use. For instance, the following Wodel program defines two blocks producing distractors for the automata exercises:

generate mutants
in "data/out/"
from "data/model/"
metamodel "http://dfaAutomaton/1.0"

with blocks {
    simple "redirects one transition" {
         modify target tar from one Transition to other State
    }
    complex "changes the finality of a state" {
         s0 = modify one State where {isFinal = true} with {isFinal = false}
         s1 = modify one State where {self <> s0} with {isFinal = true}
    }
}

The names simple and complex are the handles that the eduTest program will use below.

2. The eduTest DSL

2.1. Program structure

An eduTest program declares the domain metamodel, an optional questionnaire-level configuration, and one or more exercise definitions:

metamodel <metamodel-URI>
(navigation = free | locked)?
<exercise>+

The navigation option controls how students traverse the questionnaire: free lets them move between questions at will, while locked imposes the given order.

2.2. Exercise definitions

Every exercise follows the same pattern, regardless of its type:

<ExerciseType> (<block> (= solution)? (, <block> (= solution)?)*)? {
    <configuration>
    <test>*
}
  • <ExerciseType> is one of the eight keywords described in Section 2.3.
  • The marked blocks are the names of the Wodel blocks whose mutants populate the exercise (e.g. MatchPairs rfs1, mts2, rfs2 { ... }). Appending = solution to a block marks it as the one producing the correct answer.
  • The configuration line sets the options of the exercise (Section 2.4).
  • Each test attaches a question statement to a seed model (Section 2.5).

2.3. The exercise types

The grammar offers eight exercise constructs. Five are diagram-based:

  • AlternativeResponse: students decide whether a diagram or its textual representation is correct.
  • MultiChoiceDiagram: students select the correct diagram from several alternatives.
  • MultiChoiceEmendation: students select the textual options that correct the displayed diagram.
  • MissingWords: students fill the gaps with options from drop-down lists so that the completed text matches the statement.
  • DragAndDropText: students fill the gaps by dragging labelled text options from the available categories so that the completed text matches the statement.

And three are text-based:

  • AlternativeTextResponse: the textual counterpart of the alternative response exercise.
  • MultiChoiceText: students select the correct textual representation from several alternatives.
  • MatchPairs: students match each statement on the left with the correct option in a drop-down list.

2.4. Exercise configurations

Each exercise type uses one of three configuration layouts. All of them start with the mandatory retry = yes | no option, which states whether students may reattempt the question.

Standard configuration — used by AlternativeResponse, MultiChoiceDiagram, MissingWords and DragAndDropText:

retry = yes|no (, mode = radiobutton|checkbox)?
    (, statement = <Class> (, <Class>)*)? (, answers = <Class> (, <Class>)*)?
  • mode selects the widget used for the options: single-answer radiobutton or multiple-answer checkbox.
  • statement and answers list the metamodel classes whose elements appear, respectively, in the question statement and among the answer options.

Emendation configuration — used by MultiChoiceEmendation, which grades the selected corrections:

retry = yes|no , weighted = yes|no , penalty = <double> , order = <order> , mode = radiobutton|checkbox
    (, statement = ...)? (, answers = ...)?
  • weighted states whether each correct option contributes proportionally to the mark.
  • penalty is the mark deducted for each wrong selection (e.g. 0.0 or 0.25).
  • order arranges the options: fixed (as generated), random, options-ascending or options-descending.

Textual configuration — used by MatchPairs, MultiChoiceText and AlternativeTextResponse:

retry = yes|no (, mode = radiobutton|checkbox)? (, text = <identifier>)?
  • text names the textual representation to use, referring to an identifier defined for the model (see the %text placeholder below).

2.5. Tests: attaching statements to seed models

Each test line binds a question statement to one seed model:

description for <seed-model> = <question> (%text (('<identifier>'))? )?

The optional %text placeholder appends the textual rendering of the model to the statement; with an identifier argument, as in %text('reg-exp'), it inserts the specific representation registered under that name. For example:

MatchPairs rfs1, mts2, rfs2 {
    retry=no
    description for 'exercise6.model' = 'Select which of these options modifies
                                         the above automaton to accept only
                                         the language defined by ' %text('reg-exp')
}

3. The modelDraw DSL

3.1. Program structure

A modelDraw program states how model instances are drawn. It declares the metamodel and then, for each root class to visualise, a diagram definition listing its nodes, its relations and, optionally, extra contents:

metamodel <metamodel-URI>
<RootClass> : diagram {
    <node>*
    <relation>*
    <content>*
}

3.2. Feature conditions

Nodes and edges can be declared several times under different conditions, so the same class is drawn differently depending on its features. A condition list goes between parentheses after the class name, and each condition is a boolean feature (isInitial), a negated feature (not isFinal), a navigated feature (feat->ref), or an emptiness test (feat == null).

3.3. Nodes

<Class> (<conditions>)? (-> <TargetClass> (<conditions>)?)? : (node | markednode) (= <attribute>)?
    (compartments = { <reference>+ })?
    (shape = circle | doublecircle | record | load | logic (('<path>'))? )?
    (color = gray95)?
    (style = italic | underline)?
  • node draws a plain node labelled with the attribute given after = (by default, its name); markednode draws it highlighted, which is how initial states of automata are typically distinguished.
  • compartments nests the objects of the listed containment references inside the node, as in UML class boxes.
  • shape, color and style tune the rendering; the record, load and logic shapes serve class-like and circuit-like notations.

The automata visualisation shows the idea — the same class State is drawn in three different ways:

metamodel "http://dfaAutomaton/1.0"

Automaton: diagram {
    State(isInitial): markednode
    State(not isFinal): node shape=circle
    State(isFinal): node shape=doublecircle
    Transition(src, tar): edge label=symbol.symbol
}

3.4. Relations: edges and levels

An edge connects the objects held by two references of a class — the form used for the automata transitions above, Transition(src, tar): edge — or connects a class with a target class using the arrow form <Class> -> <TargetClass> : edge:

<Class> (<conditions>)? ( (<srcRef> (, <tarRef>)?) | -> <TargetClass> (<conditions>)? ) : edge (= <attribute>)?
    (label = <attribute-path> | { <attribute-path> (, <attribute-path>)* })?
    (src_decoration = <decoration>)? (src_label = <attribute>)?
    (tar_decoration = <decoration>)? (tar_label = <attribute>)?
  • label gives the attribute shown on the edge, possibly reached through a reference path (e.g. label = symbol.symbol), or a list of labels between braces.
  • src_decoration and tar_decoration select the arrow ends: none, triangle, diamond, odiamond, open or empty — the vocabulary needed for UML associations, aggregations, compositions and inheritance.
  • src_label and tar_label print an attribute at each end, e.g. the cardinalities of an association.

A level renders a containment reference as an edge between a container and its children, which is useful for hierarchical notations:

<Class>.<containmentRef> : edge (= <attribute>)? (label = ...)? (src_decoration = ...)? ...

3.5. Contents

A content entry customises how the values of a class are printed inside the diagram. It can map the literals of an enumerated attribute to display texts, print information reached through references, wrap an attribute between braces, or attach a fixed text symbol:

<Class> : (<attribute> [ <literal> = <text> ... ])? (<reference>(.<attribute>)?)* ({ <attribute> })? (text = <symbol>)?

For instance, a logic-circuit visualisation can map the gate-type literals to their drawing symbols:

Gate: type [AND = "and" OR = "or" NOT = "not" XOR = "xor"]

4. The modelText DSL

A modelText program defines the textual representation of the model elements — the sentences that the text-based exercises display. It declares the metamodel and, for each root class, a block of element templates:

metamodel <metamodel-URI>
<RootClass> {
    > <Class>(.<reference>)? (<conditions>)? : <word>*
    ...
}

Each template starts with >, names the class (optionally narrowed to one of its references, or guarded by the same feature conditions as modelDraw), and gives after the colon the sequence of words that verbalise it. A word is one of:

  • A constant: a literal text, e.g. final or "is the initial state".
  • A variable %<attribute> or %<reference>.<attribute>: replaced by the value of the attribute, possibly reached through a reference.
  • The macro %type: replaced by the type name of the element.

The textual representation of the automata combines all three kinds:

metamodel "http://dfaAutomaton/1.0"

>State: State %name
>State(isFinal): final
>State(isInitial): initial
>Transition: Transition %symbol.symbol
>Transition.tar: target
>Transition.src: source

Read the first three lines together: a state named q0 that is final and initial is verbalised as “State q0 final initial”, since every template whose condition holds contributes its words.

5. The mutaText DSL

A mutaText program defines the sentences that describe the applied mutations. These sentences become the options of the multiple emendation, matching-pairs, missing-words and text drag-and-drop exercises. The program declares the metamodel and a list of options, one per mutation record type:

metamodel <metamodel-URI>
> <MutationType> ((<object>))? : <valid-text> / <invalid-text>
...

Each option provides two texts separated by a slash: the valid text describes the change that corrects the mutant (the right answer), and the invalid text describes a plausible but wrong correction (a distractor). Both texts are sequences of constants and placeholder variables that Wodel-EDU fills in from the mutation registry. The optional object between parentheses restricts the option to mutations applied to a specific kind of object.

The placeholders are grouped by the mutation record they describe:

  • Any mutation: %object, %value, and the %described... family (%describedObject, %describedFromObject, %describedOldFromObject, %describedToObject, %describedOldToObject, and the First/Second variants), which print the full textual rendering of the objects as defined by modelText.
  • Attribute changes: %attName, %oldValue, %newValue.
  • Source reference changes: %refName, %fromObject, %oldFromObject.
  • Target reference changes: %srcRefName, %toObject, %oldToObject.
  • Reference swaps: %firstRefName, %firstObject, %firstFromObject, %firstToObject, %secondRefName, %secondObject, %secondFromObject, %secondToObject.
  • Attribute swaps: %firstAttName, %firstValue, %secondAttName, %secondValue.

The automata configuration illustrates the valid/invalid pattern for two mutation records:

metamodel "http://dfaAutomaton/1.0"

>TargetReferenceChanged: Change %object from %fromObject
                         to %toObject with new %refName %oldToObject /
                         Change %object from %fromObject to %oldToObject
                         with new %refName %toObject
>AttributeChanged: Change %object to %oldValue /
                   Change %object to %value

In the first option, the valid text proposes restoring the old target of the redirected transition, while the invalid text — identical in shape but with %oldToObject and %toObject swapped — proposes the change that would leave the mutant untouched, making a convincing distractor.

6. A complete specification

A full Wodel-EDU project for automata exercises combines the five files. The Wodel program of Section 1 and the modelDraw, modelText and mutaText programs of the previous sections are completed by the following eduTest questionnaire, adapted from the one published on the Wodel-EDU page:

metamodel "http://dfaAutomaton/1.0"
navigation=free
MultiChoiceDiagram simple {
    retry=no
    description for 'exercise4.model' = 'Select which of these automata
                                         accepts only "a*bab*"'
}
MultiChoiceEmendation complex {
    retry=no, weighted=no, penalty=0.0,
    order=options-descending, mode=checkbox
    description for 'exercise5.model' = 'Select the required changes so that
                                         the automaton accepts only "a+b+"'
}
MatchPairs rfs1, mts2, rfs2 {
    retry=no
    description for 'exercise6.model' = 'Select which of these options modifies
                                         the above automaton to accept only
                                         the language defined by ' %text('reg-exp')
}
MissingWords mtsrfs1 {
    retry=no
    description for 'exercise7.model' = 'Complete the following text with the
                                         options for each gap that modify this automaton
                                         to accept only the language defined by "ab(ba)*"'
}

Each exercise draws its distractors from the mutants of the referenced Wodel blocks; the diagrams are rendered as configured in modelDraw; the statements and options of the text-based exercises are built from the modelText and mutaText templates; and the generated questionnaires embed the information required for their automated correction, so the evaluation of the students' answers needs no manual intervention. The result can be deployed to the web, Moodle, Android and Apple iOS, and gamified through the integration with SGAME.
