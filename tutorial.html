Introduction

Wodel is a domain-specific language (DSL) for specifying and generating model mutants. A Wodel program declares which models to mutate, how many mutants to produce, and which mutations to apply. This page is a hands-on tutorial that walks through every part of the language, from the program header to advanced features such as blocks, cloning, retyping and OCL constraints.

All the examples on this page use the finite automata metamodel also used in the samples pages. Its main classes are Automaton, State (with attributes name, isInitial and isFinal), Transition (with references src, tar and symbol) and Symbol. The metamodel is available here: Finite Automata Ecore.

1. Anatomy of a Wodel program

Every Wodel program has the same overall shape: optional imports, a header that configures the generation, a body with the mutations (organised either as plain commands or as named blocks), and an optional constraints section:

import "..."                        // optional: reuse mutations from other files

generate 3 mutants                  // header: how many mutants...
in "data/out/"                      // ...where to store them...
from "data/model/"                  // ...which seed models to mutate...
metamodel "http://dfaAutomaton/1.0" // ...and the domain metamodel

with commands {                     // body: the mutations to apply
     modify target tar from one Transition to other State
}

constraints {                       // optional: well-formedness rules
     context State connected: "..."
}

Read it as a sentence: "generate 3 mutants in this folder, from these seed models, conforming to this metamodel, with these mutation commands". The following sections explain each part in detail.

2. The program header

The header opens with the keyword generate and admits the following elements:

generate (<number> | exhaustive)? mutants
in <output-folder>
from <seed-model-or-folder>
metamodel <metamodel-URI>
(with resources from { <name> = <path> metamodel = <URI> } , ...)?
(description <text>)?
  • generate 3 mutants produces exactly three mutants of each seed model.
  • generate mutants (no number) produces one mutant per mutation opportunity.
  • generate exhaustive mutants systematically applies the mutations in every possible way.
  • in is the output folder where the mutants are stored.
  • from is either a single seed model (e.g. "data/model/exercise1.model") or a folder (e.g. "data/model/"), in which case every model inside it is mutated.
  • metamodel is the URI of the domain metamodel that the seed models — and every generated mutant — must conform to.
  • with resources from registers additional, named model resources whose objects can be reused by the mutations (see object selection).
  • description attaches a human-readable description to the program.

A header using external resources looks like this:

generate 2 mutants
in "data/out/"
from "data/model/"
metamodel "http://dfaAutomaton/1.0"
with resources from { alphabet = "data/resources/symbols.model" metamodel = "http://dfaAutomaton/1.0" }

3. Commands and blocks

The body of a program comes in one of two flavours. The simplest is with commands { ... }: a flat list of mutations, applied to the seed models.

with commands {
     modify target tar from one Transition to other State
     remove one Transition
}

The alternative is with blocks { ... }, which organises the mutations into named stages and specifies when mutants are generated during the execution of the program. Each block has this structure:

<name> (<description>)? (from <block> (, <block>)*)? (repeat = yes|no)? {
     <mutations>
} ([<number>] | [<min>..<max>])?
  • from makes a block mutate the mutants produced by one or more earlier blocks, instead of the original seed models. This allows chaining mutation stages.
  • repeat = yes|no controls whether the block may be applied repeatedly.
  • The optional trailing cardinality states how many mutants the block generates.

The following program generates mutants in two stages: the second block mutates the models produced by the first block:

generate mutants
in "data/out/"
from "data/model/exercise1.model"
metamodel "http://dfaAutomaton/1.0"

with blocks {
    first {
         modify target tar from one Transition to other State [3]
    } [2]
    second from first repeat=no {
         modify one State with { name = random-string(4,6) }
    } [3]
}

4. Selecting objects

Almost every mutation needs to say which object(s) it operates on. Wodel provides a small set of object selection strategies that appear again and again throughout the language:

  • one State — a randomly selected object of type State.
  • one [State, Transition] — a random object of any of the listed types.
  • all State — every object of type State.
  • other State — a random State different from the one currently involved in the mutation. This is the natural choice for redirecting references.
  • s0 — a specific object previously bound to a variable (see select).
  • closure(s0) — the transitive closure of a previously selected object through a reference.
  • typed State — objects whose type is State; it is mostly used inside where conditions (e.g. relations is typed Association).
  • null — no object; useful to unset a reference or to test that a reference is empty.

Any selection can navigate references with the arrow operator -> (up to three hops), which selects the object(s) reached through the given reference:

one Transition->src          // the source state of a random transition
t0->src                      // the source state of the transition bound to t0

When the header declares external resources, a random selection can pick objects from them instead of from the seed model:

one Symbol from alphabet resources

Finally, every selection accepts an optional where { ... } filter, described next.

5. Filtering with where

The where clause restricts a selection to the objects satisfying a condition. A condition is a list of evaluations combined with and / or. There are two kinds of evaluations:

Attribute conditions compare an attribute with a value:

one State where {isInitial = true}
one State where {name <> "q0"}
one Transition where {probability >= 0.5}
one Class where {visibility in {public, private, protected, package}}
one IntLiteralExpCS where {op in ['add', 'subtract', 'multiply', 'divide']}

The available comparison operators are =, <> (different), >, >=, <, <=, in (membership in a list), is and not. The compared value can also be an aggregate such as min(attribute) or max(attribute), or an attribute of a previously selected object:

t0 = select one Transition where {probability = min(probability)}

Reference conditions compare a reference — or the special keywords self (the candidate object itself) and container (its container) — with another object selection or with null:

remove all Transition where {src = null}
one State where {self <> s0}
one Transition where {self <> t0 and src = t0->src}
one ClassDiagram where {relations is typed Association}

6. The mutation primitives

Wodel provides nine mutation primitives: object creation, object deletion, reference creation, reference deletion, reference redirection (source and target), feature modification, object cloning and object retyping. Together with object selection and composite mutations, they cover the mutations found in the mutation-testing literature. Each primitive is presented below with its syntax skeleton and an example. In every skeleton, <selection> stands for any of the strategies of Section 4, and the optional trailing [<cardinality>] is explained in Section 8.

6.1. Object creation — create

(<var> =)? create <Type> (in <selection>)? (with { <initialisations> })? ([<cardinality>])?

Creates a new object of the given type, optionally inside a specific container, and optionally initialising its attributes and references (see Section 7). The result can be bound to a variable for later use:

s = create State with {isFinal = true}
create Transition with {tar = s, symbol = one Symbol}

6.2. Object deletion — remove

remove <selection> (from <selection>)? ([<cardinality>])?

remove one State where {isInitial = false}
remove all Transition where {src = null}

6.3. Reference creation — create reference

(<var> =)? create reference <ref> (to <selection>)? (in <selection>)? ([<cardinality>])?

Creates a new reference of the given kind, from the object selected with in to the object selected with to:

create reference symbol to one Symbol in one Transition

6.4. Reference deletion — remove reference

remove one reference <ref> in <Type> ([<cardinality>])?   // a random reference
remove reference <ref> in <selection> ([<cardinality>])?  // in a specific object
remove all reference <ref> in <Type>                       // every such reference

remove one reference symbol in Transition

6.5. Reference redirection — modify source / modify target

modify source <ref> (from <selection>)? (to <selection>)? ([<cardinality>])?
modify target <ref> (from <selection>)? (to <selection>)? ([<cardinality>])?

Redirects the source or target end of a reference. The from selection identifies the reference owner, and to gives the new end. This is the mutation of the classical example on the home page:

modify target tar from one Transition to other State

6.6. Feature modification — modify

(<var> =)? modify <selection> (with { <changes> })? ([<cardinality>])?

Modifies the attributes and references of the selected object(s), using the constructs of Section 7:

s0 = modify one State where {isInitial = true} with {isInitial = false}
s1 = modify one State where {self <> s0} with {isInitial = true}

6.7. Object selection — select

<var> = select <selection> (in <selection>)?
<var> = select sample from <selection> (with (equals | distinct) { <features> })?

select binds an object to a variable without mutating it, so that later mutations can refer to it. Its select sample variant picks a set of objects whose listed features are all equals or all distinct:

t0 = select one Transition where {probability = min(probability)}
t1 = select one Transition where {self <> t0 and src = t0->src}
sts = select sample from all State with distinct { name }

6.8. Object cloning — clone

(<var> =)? (deep)? clone <selection> (in <selection>)? (with { <changes> })? ([<cardinality>])?

Clones the selected object, optionally placing the copy in a given container and modifying its features. The deep modifier also clones the contained objects:

clone one State with {name = catend("_copy")}
deep clone one Automaton

6.9. Object retyping — retype

(<var> =)? retype <selection> (in <selection>)? as (<Type> | [<Type>, ...]) (with { <changes> })? ([<cardinality>])?

Changes the type of an object to one of its sibling types (i.e. types sharing a common superclass), preserving its compatible features. If a list of types is given, one is chosen at random:

retype one Class as AssociationClass
retype one [Association, Aggregation, Composition, Dependency]

6.10. Composite mutations

(<var> =)? [ <mutation> <mutation> ... ] ([<cardinality>])?

A composite mutation groups several mutations between square brackets so they are applied together as a unit, optionally with a shared cardinality:

[
     s = create State with {isFinal = true}
     create Transition with {tar = s, symbol = one Symbol}
] [1..2]

7. Setting attributes and references — the with { ... } clause

The with { ... } clause of create, modify, clone and retype holds a comma-separated list of attribute and reference changes.

Attribute values

The right-hand side of an attribute assignment can be:

  • A literal: isFinal = true, name = "q0", counter = 3, probability = 0.5.
  • A value from a list, chosen at random: op in ['add', 'subtract', 'multiply', 'divide'] for strings, or visibility in {public, private, protected, package} for enumeration literals.
  • A random value:
    • name = random-string(1, 4) — a random string between 1 and 4 characters long;
    • counter = random-int(0, 10) — a random integer in the given range;
    • probability = random-double(0.0, 1.0) — a random double in the given range;
    • id = random-int-string(1, 4) — a random numeric string;
    • isFinal = random-boolean — a random boolean;
    • probability = random(0.0, t0.probability) — a random number bounded by another object's attribute;
    • value = random — a random value of the attribute's type.
  • A string transformation of the current value: name = upper, name = lower, name = catstart("pre_"), name = catend("_post"), name = replace("old", "new").
  • An aggregate over the attribute: probability = min(probability), probability = max(probability).
  • Another object's attribute: name = s0.name.

Attribute helpers

Besides plain assignment, four helper operations are available:

unset(name)                 // clears the attribute
reverse(isFinal)            // flips a boolean attribute
swap(name, s0.name)         // exchanges two attribute values
copy(name, s0.name)         // copies an attribute value

Arithmetic updates

Numeric attributes can be updated in place with the arithmetic operators +, -, *, / and %, as in the probabilistic automata samples:

modify t1 with {probability += random(0.0, t0.probability)}
modify t2 with {probability -= t1.probability}

Reference changes

tar = s                     // sets the reference to a selected object
states += s                 // adds an object to a multi-valued reference
states -= s                 // removes an object from a multi-valued reference
swapref(src, tar)           // exchanges two references
unsetref(symbol)            // clears the reference
symbol.name = "a"           // sets an attribute of the referenced object

For example, the following mutation reverses the direction of an association by swapping its ends:

modify one Association with {swapref(src, tar)}

8. Cardinalities

Most mutations, composites and blocks accept a trailing cardinality between square brackets, stating how many times they are applied (or, for blocks, how many mutants they generate):

remove one Transition [2]        // applied exactly twice
create Transition [1..3]         // applied between one and three times
remove one Transition [1..*]     // applied one or more times

9. Imports and libraries

Wodel supports the modular reuse of mutations. A file can be declared as a library — a collection of mutations for a metamodel, without a generation header:

library for metamodel "http://dfaAutomaton/1.0"

with commands {
     modify target tar from one Transition to other State
}

Other programs can then reuse it with an import statement placed before the header:

import "automata-mutations.mutator"

generate 2 mutants
in "data/out/"
from "data/model/"
metamodel "http://dfaAutomaton/1.0"

with commands {
     remove one Transition
}

10. Constraints

For every generated mutant, the Wodel engine checks that the result is a valid model: it must conform to the domain metamodel and satisfy its integrity constraints. In addition, the optional constraints { ... } section at the end of the program lets you define extra well-formedness rules that every mutant must satisfy. Each rule declares a context type, an identifier, and either an OCL-like expression (introduced by :) or the name of a Java class implementing the check (introduced by ::):

constraints {
    context State connected: "isInitial or Set{self}->
            closure(s | Transition.allInstances()->
            select(t | t.tar=s)->collect(src))->exists(s | s.isInitial)"
    context Automaton deterministic:: "AutomatonChecker"
}

The expression language is a compact OCL subset. Expressions navigate features with . and collections with ->, compare values with = and <>, combine conditions with and / or, and support the iterators collect, exists, forAll and iterate (with accumulator variables introduced by $). Mutants violating a constraint are discarded.

11. A complete example

The following program brings everything together. It generates up to three mutants of an automaton model in two chained blocks, keeps a variable across mutations, and enforces that every state remains reachable:

generate mutants
in "data/out/"
from "data/model/exercise1.model"
metamodel "http://dfaAutomaton/1.0"
description "Mutations for automata exercises"

with blocks {
    structure "changes the accepted language" {
         modify target tar from one Transition to other State
         s = create State with {isFinal = true, name = random-string(1, 4)}
         create Transition with {tar = s, symbol = one Symbol}
    } [2]
    naming "renames states" from structure repeat=no {
         modify one State with {name = random-string(4, 6)}
    } [3]
}
constraints {
    context State connected: "isInitial or Set{self}->
            closure(s | Transition.allInstances()->
            select(t | t.tar=s)->collect(src))->exists(s | s.isInitial)"
}

Step by step: the structure block redirects a transition and adds a new final state reachable through a fresh transition, producing two mutants; the naming block takes those mutants and renames a random state in each, producing three further mutants each; and the constraint discards any mutant containing an unreachable state.
